## 1. 背景

Pharos 要解决的不是“再做一个任务列表”，而是把真实工作中不断散落的信号转成可以推进、可以中断、可以 review、可以继续执行的工作流。

当前目标场景集中在以下内部工作系统：

1. 飞书聊天：群消息、私聊、@、thread、转发给机器人或应用的消息。
2. 飞书项目：需求、任务、缺陷、评论、状态变更、指派、阻塞。
3. 私有部署 GitLab：Merge Request、Issue、讨论、review 请求、pipeline 状态、代码变更上下文。
4. 飞书文档：文档评论、@、相关需求文档、方案文档、上下文资料。

MVP 的核心不是把这些系统分别做深，而是先把两层模型打稳。

用户心智模型：

```text
看住信号 -> 整理证据 -> 准备动作 -> 等我拍板 -> 执行留痕
```

内部实现模型：

```text
收集信号 → 归一化为工作请求 → 识别是否需要行动 → 补充上下文 → 自动执行安全步骤 → 遇到门禁等待 review → 用户批准/编辑/拒绝 → 继续执行 → 留下证据和反馈
```

后续新增数据源、新增技能、新增执行器，都应该挂到这套模型上，而不是让每个集成都变成一条野生水管。

## 2. 产品定位

Pharos 是一个 local-first 的 AI 工作驾驶舱。

它看住工作系统里的信号，识别真正需要用户或 agent 处理的工作请求，整理证据、准备下一步，并在需要用户判断、授权或承担责任时停下来等待 review。

### 2.1 一句话

Pharos watches your work systems, prepares the next move, and asks before it acts.

### 2.2 中文描述

Pharos 帮你看住工作系统，整理证据、准备下一步；真正行动前一定问你。

### 2.3 GitHub description 推荐

推荐使用：

```text
Pharos watches your work systems, prepares the next move, and asks before it acts.
```

备选：

```text
Local-first AI work cockpit for review-gated work automation.
```

### 2.4 User Mental Model / 用户心智

用户不应该感觉自己在管理一张内部状态表，而是在看一个按注意力和决策压力整理好的工作驾驶舱：

```text
看住信号 -> 整理证据 -> 准备动作 -> 等我拍板 -> 执行留痕
```

面向用户的主分组是：

| 分组 | 用户要回答的问题 | 产品含义 |
|---|---|---|
| Needs Decision | 现在有什么需要我拍板？ | Pharos 已经准备好证据和建议动作，需要用户批准、编辑、拒绝或归档。 |
| Needs Input | 现在还缺我补什么？ | Pharos 缺上下文、权限、判断或选择，不能继续安全推进。 |
| Watching | Pharos 正在看住或准备什么？ | 新信号、triage、只读分析、草稿生成、等待外部信息等后台推进中的事项。 |
| Handled | 今天已经处理了什么？ | 已完成、已写回、已归档或用户已明确处理的事项。 |
| Noise | 哪些被过滤掉了？ | 被识别为无行动、误报、重复或用户主动归档的信号。 |

内部仍然需要保留精确状态机、风险等级、Review Gate、Evidence 和 Timeline。区别是：内部状态用于实现和追踪，用户首屏语言用于降低认知负担。

## 3. MVP 目标

MVP 要证明一件事：

> Pharos 能在真实工作流中稳定完成“看住信号、整理证据、准备动作、等我拍板、执行留痕”这一套闭环，并用内部状态机可靠支撑它。

### 3.1 成功后的体验

用户每天打开 Pharos 后，应该能看到：

1. 哪些事情需要我现在拍板。
2. 哪些事情还缺我的输入或授权。
3. 哪些事情 Pharos 正在看住、整理或准备。
4. 每件事从哪里来，为什么被带到我面前。
5. 系统已经读了哪些上下文，依据哪些证据做了什么判断。
6. 我批准后它会对哪个外部系统做什么。
7. 完成后可以追溯结果、证据、时间线和用户决策。

### 3.2 MVP 不追求的体验

MVP 不追求“全自动处理所有工作”。

第一版的正确目标是：

```text
该自动的自动推进；该停的必须停住；该用户判断的地方给足证据。
```

## 4. 产品形态

### 4.1 主形态：macOS App

MVP 建议做成 macOS 桌面应用，而不是 Web 应用。

原因：

1. 用户日常工作环境主要在 macOS。
2. Pharos 需要常驻、通知、菜单栏、快捷捕获、后台同步和本地凭证管理。
3. 第一版只服务个人，local-first 更符合隐私、安全和迭代速度。
4. 它需要和本地代码仓库、CLI、技能仓库、私有 GitLab、飞书配置等工作环境打交道。
5. 后续若产品化，可以再拆云端 relay、团队后台或 Web 控制台。

### 4.2 MVP Surface

| Surface | 定位 | P0 要求 |
|---|---|---|
| macOS 主窗口 | 每日工作驾驶舱，处理请求和 review | 必须 |
| 菜单栏入口 | 待 review 数、快速打开、快速捕获 | 必须 |
| 本地通知 | 只提醒高优先级和等待 review 的事项 | 必须 |
| 快速捕获 | 手动把一段文本、链接、想法转成请求 | 必须 |
| CLI/本地投递入口 | 让脚本、`vc` 或其他工具把事件投进 Pharos | 建议 P0 |
| 云端服务 | webhook relay、团队协作、跨设备同步 | 不做 |

### 4.3 与现有项目关系

Pharos 是新项目，不直接塞进 `Nyx`，也不依赖 `intent-to-evidence`、`skills`、`agent-skill-control-theory` 等项目的当前实现。

现有项目在 MVP 中只作为思想和未来集成方向：

| 项目 | MVP 关系 |
|---|---|
| `vc` | 可作为未来 CLI 入口或本地工作流桥接参考；MVP 不强依赖。 |
| `nyx` | 可参考桌面应用经验；不作为 Pharos 的产品容器。 |
| `skills` | 可参考 skill 形态；MVP 先定义 skill 接口和门禁模型，不要求直接接入。 |
| `intent-to-evidence` | 可参考“意图到证据”思想；当前不作为实现依赖。 |
| `agent-skill-control-theory` | 可参考 skill 控制理论；当前不要求实现完整理论框架。 |
| `agent-operable-architecture` | 可作为未来代码自动化和风险边界参考；MVP 只保留 evidence / review / safety 思路。 |

原则：

> Pharos MVP 先把流程和产品模型做稳。现有理论和工具成熟后，再作为插件、技能、runtime 或证据协议接入。

## 5. MVP 范围

### 5.1 P0 数据源

MVP 只围绕以下 4 类数据源：

1. 飞书聊天。
2. 飞书项目。
3. 私有部署 GitLab。
4. 飞书文档。

### 5.2 P0 核心能力

MVP 必须包含：

1. 统一收集入口：从 P0 数据源和手动/CLI 入口收集原始工作信号。
2. 需求识别：判断信号是否构成需要处理的工作请求。
3. 归一化模型：把不同来源的信号转成统一的工作请求。
4. 去重与合并：同一事项的多条消息、评论、状态变化应合并，而不是刷屏。
5. 上下文整理：读取相关聊天、项目、文档、MR、Issue 等上下文，形成可 review 的摘要。
6. 自动执行：自动完成只读分析、摘要、草稿、建议动作、局部检查等安全步骤。
7. Review Gate：外部可见、高风险、低置信度或需要用户判断的动作必须停下来。
8. 继续执行：用户批准或编辑后批准，系统继续执行被授权的动作。
9. 证据链：每个结论、草稿、外部动作都能追溯来源和过程。
10. 反馈闭环：用户的批准、编辑、拒绝、误报标记会影响后续规则。
11. 本地指标：记录这套流程是否真的减少工作负担。

### 5.3 明确非目标

MVP 不做：

1. 通用工作流编排器。
2. 插件市场。
3. 多用户团队版。
4. 云同步。
5. 移动端。
6. 通用聊天机器人。
7. 自动生产写操作。
8. 自动合并代码。
9. 自动发布上线。
10. 个人生活事务处理。
11. 所有第三方 SaaS 全量集成。
12. 对现有 `intent-to-evidence`、`skills` 等项目的强绑定。

## 6. 核心概念模型

本 PRD 只定义产品概念，不定义具体代码结构、数据库 schema 或技术栈。

### 6.1 Source Signal，来源信号

外部系统里发生的一条原始信号。

例子：

- 飞书群里有人 @ 用户问一个问题。
- 飞书项目里一个需求被指派给用户。
- GitLab MR 请求用户 review。
- GitLab discussion 里有人 mention 用户。
- 飞书文档评论里有人 @ 用户。
- 用户手动把一段文字或链接投递给 Pharos。

来源信号必须保留原始来源和可追溯链接。

### 6.2 Work Request，工作请求

Pharos 识别后认为“需要处理”的事项。

工作请求不是普通 todo，而是一个可以被系统推进的对象。它应该包含：

- 它来自哪里。
- 它在请求什么。
- 为什么需要进入 Pharos。
- 当前是否需要用户参与。
- 系统已经做了什么。
- 下一步可以做什么。
- 风险在哪里。
- 证据是什么。

### 6.3 Context Bundle，上下文包

系统为理解一个工作请求而收集的有限上下文。

例子：

- 相关飞书 thread。
- 飞书项目字段和历史评论。
- GitLab MR diff 摘要、discussion、pipeline 状态。
- 飞书文档的相关段落。
- 用户手动补充的说明。

上下文包必须有边界，不能无限抓取。

### 6.4 Proposed Action，建议动作

系统认为可以执行或建议用户执行的动作。

例子：

- 生成一条飞书回复草稿。
- 生成一个项目评论草稿。
- 生成 MR review 结论。
- 读取文档并提取决策点。
- 更新本地卡片状态。
- 向外部系统写一条评论。
- 请求用户补充上下文。

建议动作必须带有风险等级和是否需要审批。

### 6.5 Skill，执行技能

Skill 是能处理某类工作请求的执行能力。

MVP 中的 skill 可以非常简单，不要求接入完整技能仓库。它可以是内置 prompt、规则、脚本、CLI、模型调用或后续独立插件。

Skill 必须声明：

- 能处理什么类型的工作请求。
- 需要哪些上下文。
- 会产生哪些建议动作。
- 是否可能写外部系统。
- 默认风险等级。
- 输出需要哪些证据。

### 6.6 Review Gate，审批门禁

当系统准备执行需要用户判断或授权的动作时，必须停在 Review Gate。

Review Gate 不是简单的 approve 按钮。它是用户接管控制权的地方。

用户可以：

- 批准。
- 编辑后批准。
- 拒绝。
- 要求补充上下文。
- 稍后处理。
- 归档。
- 标记误报。
- 禁止同类自动化。

### 6.7 Evidence，证据

Evidence 是用户判断和系统复盘的依据。

证据可以来自：

- 原始消息。
- 项目字段。
- MR diff 摘要。
- 文档段落。
- 评论记录。
- pipeline 状态。
- skill 输出。
- 执行结果。
- 用户决策记录。

所有进入 Review Gate 的动作都必须有证据。

### 6.8 Run，执行过程

Run 是系统对一个工作请求执行的一次处理过程。

Run 应记录：

- 触发原因。
- 读取了哪些上下文。
- 选择了哪些技能。
- 产生了哪些建议动作。
- 是否中断等待 review。
- 用户做了什么决定。
- 后续执行是否成功。

## 7. 核心流程

### 7.1 总流程

```text
1. Capture
   从飞书聊天、飞书项目、GitLab、飞书文档或手动入口收集来源信号。

2. Normalize
   将来源信号归一化，保留来源、actor、时间、链接、文本、可见范围。

3. Detect
   判断它是不是工作请求。
   不是请求则归档为噪音或背景信号。

4. Merge
   判断是否属于已有工作请求。
   是则更新已有请求，不重复创建。

5. Classify
   判断请求类型、优先级、风险等级、是否需要用户。

6. Gather Context
   按需读取相关聊天、项目、MR、Issue、文档上下文。

7. Plan / Execute Safe Steps
   执行只读分析、摘要、草稿生成、建议下一步等安全步骤。

8. Gate
   如果动作需要用户判断或授权，停在 Review Gate。

9. User Review
   用户批准、编辑、拒绝、归档、要求补上下文或标记误报。

10. Continue
   如果用户批准，系统继续执行被授权动作。

11. Complete
   记录结果、证据、指标和反馈。
```

### 7.2 请求状态

MVP 至少需要支持以下状态：

| 状态 | 含义 |
|---|---|
| New | 刚创建，尚未处理。 |
| Triaging | 正在判断是否需要行动、如何分类。 |
| Needs Context | 信息不足，需要系统或用户补上下文。 |
| Running | 系统正在自动执行安全步骤。 |
| Ready for Review | 已推进到需要用户 review。 |
| Waiting | 暂停等待外部信息、时间或用户后续处理。 |
| Approved | 用户已批准某个动作，等待继续执行。 |
| Executing | 正在执行用户批准后的动作。 |
| Done | 请求完成。 |
| Failed | 执行失败，需要查看原因或重试。 |
| Snoozed | 用户延后处理。 |
| Archived | 用户归档或系统判断无需处理。 |

面向用户展示时，不直接把这些状态当作 Today 的主导航。Today 使用用户表层分组，内部状态通过映射支持实现和筛选：

| 用户表层分组 | 用户问题 | 内部状态 / 对象映射 |
|---|---|---|
| Needs Decision | 我现在要批准、编辑、拒绝还是归档？ | Ready for Review、带有 reviewable Proposed Action 的请求、需要用户承担责任的 L3+ 动作。 |
| Needs Input | Pharos 还需要我补什么？ | Needs Context、Failed 中的信息不足、缺权限、缺判断或用户待确认字段。 |
| Watching | Pharos 正在看住或准备什么？ | New、Triaging、Running、Waiting、Snoozed、Approved、Executing。 |
| Handled | 今天已经处理了什么？ | Done、已执行外部写回、用户已本地完成、已明确关闭的请求。 |
| Noise | 哪些被过滤或归档了？ | Archived、误报、重复信号、无行动背景信号。 |

这个映射是产品和实现之间的合同：用户看到的是注意力分组；系统仍然保留精确状态流转、风险策略和 Timeline。

### 7.3 风险等级

MVP 使用风险等级控制自动化边界。

| 等级 | 含义 | MVP 策略 | 例子 |
|---|---|---|---|
| L0 只读 | 只读取和整理信息，不改变外部系统。 | 自动执行。 | 总结 thread、读取 MR、整理文档上下文。 |
| L1 私人产物 | 生成只有用户可见的草稿或建议。 | 自动执行。 | 回复草稿、review 草稿、项目评论草稿。 |
| L2 本地写入 | 只改变 Pharos 本地状态。 | 自动执行或轻审批。 | 归档、打标签、记录规则命中。 |
| L3 外部可见写入 | 会写入飞书、GitLab、飞书项目、飞书文档等外部系统。 | 必须 Review Gate。 | 发群回复、发 MR comment、写项目评论、写文档评论。 |
| L4 高影响动作 | 可能改变代码、项目承诺、发布、线上状态。 | MVP 不自动执行，只生成建议。 | 提交代码、创建 MR、改配置、触发发布。 |
| L5 禁止动作 | 不允许自动化或风险不可接受。 | 禁止执行。 | 敏感权限、财务/法务/人事承诺、生产危险操作。 |

硬规则：

> 任何 L3 及以上动作，都不能在没有用户批准的情况下执行。

## 8. P0 数据源需求

### 8.1 飞书聊天

目标：从飞书聊天里识别真正需要用户处理的请求。

P0 收集范围：

- 用户配置的群。
- 用户配置的私聊或机器人可见会话。
- @ 用户的消息。
- 明确点名用户的消息。
- 用户转发给 Pharos 的消息。
- 命中用户配置关键词的消息。

P0 处理能力：

- 判断消息是否需要行动。
- 合并同一 thread 的连续消息。
- 提取对方在请求什么。
- 生成回复草稿或下一步建议。
- 识别信息型消息、寒暄、确认类消息并归档。
- 需要回复外部消息时进入 Review Gate。

验收标准：

- 在配置群里 @ 用户并提出问题后，Pharos 能创建或更新一个工作请求。
- 同一 thread 的连续补充不会生成多张重复请求。
- “收到”“谢谢”“已处理”这类无行动消息默认不进入 Today 主队列。
- 每个聊天请求都能看到原消息、会话名、发送者、时间、上下文摘要。
- 发送飞书回复必须经过用户批准。

### 8.2 飞书项目

目标：从飞书项目中识别与用户相关的需求、任务、缺陷和阻塞。

P0 收集范围：

- 分配给用户的项目项。
- 评论中 @ 用户或明确等待用户回复的项目项。
- 状态变为阻塞、待确认、待评审、待验收等用户关心状态的项目项。
- 用户负责范围内临近截止日期或长期未推进的项目项。
- 用户手动添加到关注范围的项目项。

P0 处理能力：

- 总结项目项当前状态。
- 识别是否需要用户行动。
- 识别阻塞点、待决策点、缺失信息。
- 生成项目评论草稿、推进建议或拆分建议。
- 更新外部项目评论或状态前必须进入 Review Gate。

验收标准：

- 一个分配给用户的测试项目项被同步后，Pharos 能创建工作请求。
- 项目评论中 @ 用户后，Pharos 能创建或更新请求。
- 请求详情中能看到项目项链接、状态、负责人、优先级、最近评论摘要。
- 系统能给出“为什么这件事需要我处理”的解释。
- 写飞书项目评论或改状态必须经过用户批准。

### 8.3 私有部署 GitLab

目标：从私有 GitLab 中识别需要用户参与的代码和研发协作请求。

P0 收集范围：

- 请求用户 review 的 Merge Request。
- 用户被 mention 的 MR、Issue、discussion。
- 用户负责或关注的 MR 出现新 discussion、pipeline 失败或需要处理的反馈。
- 分配给用户的 Issue。
- 用户手动添加关注的 repo、MR、Issue。

P0 处理能力：

- 总结 MR / Issue 当前状态。
- 对 MR 进行只读分析：变更摘要、风险点、测试缺口、需要人工判断的地方。
- 生成 review comment 草稿或 issue 回复草稿。
- 将 pipeline 状态、discussion、相关链接作为证据。
- 任何 GitLab 外部写入都必须进入 Review Gate。

验收标准：

- 一个测试 MR 请求用户 review 后，Pharos 能创建或更新工作请求。
- MR 请求详情中能看到标题、作者、链接、变更摘要、风险摘要、pipeline 状态和建议 review 草稿。
- MR 有新 commit 或 discussion 时，Pharos 更新同一请求，而不是重复刷屏。
- 发表 MR comment、Issue comment、review 结论必须经过用户批准。
- MVP 不自动提交代码、不自动创建 MR、不自动合并 MR。

### 8.4 飞书文档

目标：让文档既可以作为请求来源，也可以作为上下文来源。

P0 收集范围：

- 文档评论中 @ 用户。
- 用户手动分享或转发给 Pharos 的文档链接。
- 飞书聊天或项目项里引用的文档。
- 用户配置为关注的文档或文件夹。

P0 处理能力：

- 识别文档评论是否需要用户行动。
- 提取文档中的相关段落作为上下文。
- 总结需求背景、方案要点、待确认点。
- 生成文档评论草稿或反馈草稿。
- 写文档评论或修改文档内容前必须进入 Review Gate。

验收标准：

- 文档评论 @ 用户后，Pharos 能创建或更新工作请求。
- 用户手动投递一个文档链接后，Pharos 能生成摘要和待确认点。
- 当聊天、项目、GitLab 请求引用文档时，Pharos 能把文档摘要加入证据区。
- 写文档评论必须经过用户批准。
- MVP 不自动改文档正文。

## 9. 核心功能需求

### 9.1 Daily Cockpit

目标：用户每天只看一个地方，就知道现在什么需要注意、什么需要决定、什么可以放心交给 Pharos 看住。

Today 页面至少包含：

1. Needs Decision：等待用户批准、编辑、拒绝、归档或稍后处理的请求。
2. Needs Input：缺少上下文、权限、判断或用户选择，Pharos 无法继续安全推进的请求。
3. Watching：Pharos 正在看住、整理、分析、生成草稿或等待外部变化的请求。
4. Handled：今日已完成、已写回、已本地处理或已明确关闭的请求。
5. Noise：被过滤、归档、判定为误报或重复的信号摘要。

验收标准：

- Needs Decision 永远在最上方。
- 每张请求列表项必须展示来源、标题、为什么出现、建议下一步、优先级、风险等级、更新时间。
- 涉及外部写入的请求，列表项必须展示“批准后会写到哪里”的简要说明。
- Waiting、Running、Ready for Review 等内部状态可以作为标签或筛选条件出现，但不能成为 Today 的主要心智。
- 用户能按来源、状态、风险、优先级筛选。
- 用户能一键打开原始来源。

### 9.2 Request Detail

目标：让用户在 10 秒内判断这件事是什么、为什么进来、现在要不要拍板、批准后会发生什么。

详情页必须先回答四个问题：

1. What is this? 这是什么请求。
2. Why did Pharos bring it to me? 为什么 Pharos 把它带到我面前。
3. What evidence did Pharos use? Pharos 使用了哪些证据。
4. What exactly will happen if I approve? 我批准后具体会发生什么。

详情页必须展示：

- 原始来源和链接。
- 请求摘要。
- 为什么被识别为工作请求。
- 用户表层分组和内部状态。
- 优先级和风险等级。
- 系统已读取的上下文。
- 系统已执行的步骤。
- 建议动作。
- 草稿或输出内容。
- 证据区。
- 执行时间线。
- Review Gate 操作区。
- 批准后写入目标、写入内容、权限和风险说明。

验收标准：

- 用户不打开外部系统，也能理解大多数低/中复杂度请求。
- 用户能在 Review Gate 中看到动作会写到哪个系统、哪个对象、以什么内容写入。
- 用户可以复制草稿。
- 用户可以编辑草稿后批准。
- 用户可以要求补充上下文。
- 用户可以标记误报并归档。

### 9.3 Source Management

目标：管理数据源连接、收集范围和写入权限。

每个数据源需要展示：

- 是否启用。
- 连接状态。
- 最近同步时间。
- 最近错误。
- 收集范围。
- 写入权限是否开启。
- 默认风险策略。

验收标准：

- 任意单个 source 失败不会影响其他 source 和本地 app。
- 用户可以禁用某个 source。
- 用户可以限制飞书群、项目、repo、文档范围。
- 外部写入权限默认关闭。
- 即使开启写入权限，L3 动作仍必须审批。

### 9.4 Demand Detection / Triage

目标：把“所有信号”筛成“需要处理的工作请求”。

Triage 至少要输出：

- 是否创建/更新工作请求。
- 请求类型。
- 优先级。
- 风险等级。
- 为什么需要用户或系统处理。
- 下一步建议。
- 是否需要补上下文。
- 是否需要通知用户。

验收标准：

- 每个进入 Today 的请求都有“进入原因”。
- 无行动信号不会进入主队列。
- 同一事项不会重复刷屏。
- 用户标记误报后，系统记录反馈。
- MVP 可以先用规则 + 模型判断，不要求自动学习长期偏好。

### 9.5 Context Gathering

目标：为工作请求补充足够但有边界的上下文。

上下文来源包括：

- 当前来源的原始内容。
- 同一 thread 或评论链。
- 相关飞书项目项。
- 相关飞书文档。
- 相关 GitLab MR / Issue / discussion。
- 用户手动补充的信息。

验收标准：

- 请求详情中能看到系统读取了哪些上下文。
- 上下文摘要必须区分事实、推断和待确认点。
- 如果上下文不足，系统应进入 Needs Context，而不是强行执行。
- 用户可以手动补充上下文并重新运行。

### 9.6 Skill Routing / Execution

目标：根据工作请求类型选择合适技能推进。

MVP 不要求完整插件系统，但必须保留扩展边界。

P0 至少需要内置以下技能能力：

1. 需求识别与整理：把来源信号转成工作请求摘要。
2. 上下文摘要：整理聊天、项目、文档、MR/Issue 关键点。
3. 回复/评论草稿：为飞书、项目、GitLab、文档生成回复草稿。
4. GitLab MR 只读 review：生成变更摘要、风险点、测试缺口和 review 草稿。
5. 项目推进建议：根据项目项状态生成下一步建议或评论草稿。
6. 文档理解：提取背景、决策点、待确认点。

验收标准：

- 至少 3 类工作请求能被自动推进到草稿或建议动作阶段。
- skill 输出必须带有对应证据。
- skill 不能绕过风险策略直接写外部系统。
- skill 失败时，请求进入 Failed 或 Needs Context，并展示错误原因。

### 9.7 Review Gate

目标：所有需要用户判断或授权的动作都在这里停住。

Review Gate 支持操作：

- Approve。
- Edit and Approve。
- Reject。
- Request More Context。
- Snooze。
- Archive。
- Mark False Positive。
- Disable Similar Automation。
- Open Source。
- Copy Draft。

验收标准：

- 所有 L3 动作必须等待用户批准。
- 用户批准后才能继续执行。
- 用户编辑后批准，最终执行内容必须是编辑后的版本。
- 用户拒绝后，不得执行该动作。
- 用户要求补上下文后，系统创建新的执行过程并追加证据。
- 每次用户决策都必须可追溯。

### 9.8 Controlled Writeback

目标：让系统可以在授权后对外部系统写入，但不能越权。

MVP 允许的写入类型：

- 发送飞书聊天回复。
- 写飞书项目评论。
- 写 GitLab MR / Issue comment 或 review discussion。
- 写飞书文档评论。
- 更新 Pharos 本地状态。

MVP 禁止的写入类型：

- 自动修改代码。
- 自动创建或合并 MR。
- 自动修改飞书文档正文。
- 自动改线上配置。
- 自动触发发布。
- 自动做不可逆或高影响操作。

验收标准：

- 外部写入必须关联用户批准记录。
- 每次写入必须在时间线中留下结果。
- 写入失败必须可见且可重试。
- 未批准写入次数必须永远为 0。

### 9.9 Evidence & Timeline

目标：用户能知道系统为什么这么判断、执行了什么、结果是什么。

每个等待 review 的请求必须有：

- 原始来源。
- 上下文摘要。
- 关键证据。
- 系统判断摘要。
- 建议动作。
- 风险说明。
- 执行过程时间线。

验收标准：

- 用户能回答“这个结论从哪里来”。
- 用户能回答“批准后它会写到哪里”。
- 用户能回答“系统已经做了什么，还没做什么”。
- 失败不会静默吞掉，必须出现在请求详情里。

### 9.10 Feedback & Rules

目标：让 Pharos 越用越贴合用户，但 MVP 不要求复杂学习系统。

P0 支持：

- 标记误报。
- 标记漏报。
- 标记优先级错误。
- 禁止某类来源进入 Today。
- 某关键词提升优先级。
- 某群/项目/repo 降低优先级。
- 某类动作永远需要 review。

验收标准：

- 用户能创建、启用、禁用、删除简单规则。
- 请求详情能展示命中了哪些规则。
- 规则变更后，后续 triage 会受影响。
- 误报反馈能在 metrics 中看到。

### 9.11 Metrics

目标：判断 MVP 是否真的有用。

P0 本地指标：

- 每日来源信号数。
- 每日工作请求数。
- 每日进入 Today 数。
- 每日 Needs Decision 数。
- 自动推进到草稿/建议动作的比例。
- 用户批准率。
- 编辑后批准率。
- 拒绝率。
- 误报数。
- 归档数。
- 平均从信号到请求的时间。
- 平均从请求到 Ready for Review 的时间。
- 用户处理请求用时。
- 外部写入次数。
- 未审批外部写入次数。

验收标准：

- Metrics 页面能查看最近 7 天数据。
- 指标可以导出为本地 JSON 或 Markdown。
- 未审批外部写入次数必须始终为 0。
- 每个 source 的信号量、请求转化率、误报率可查看。

## 10. 端到端用户故事

### 10.1 飞书聊天请求

1. 同事在配置群里 @ 用户，询问一个项目或技术问题。
2. Pharos 收到来源信号。
3. 系统判断这是一个需要回复的工作请求。
4. 系统拉取 thread 上下文和相关文档/项目链接。
5. 系统生成摘要和回复草稿。
6. 请求进入 Ready for Review。
7. 用户编辑后批准。
8. Pharos 发送飞书回复。
9. 请求进入 Done，并记录证据和执行结果。

验收标准：

- 用户能在 Pharos 内完成理解、编辑、批准和发送。
- 未批准前不会发送飞书消息。

### 10.2 飞书项目推进请求

1. 一个项目项被指派给用户，或评论中 @ 用户要求确认。
2. Pharos 创建或更新工作请求。
3. 系统整理项目状态、历史评论、关联文档。
4. 系统识别待决策点或阻塞点。
5. 系统生成下一步建议和评论草稿。
6. 用户批准或要求补上下文。
7. 批准后，Pharos 写入项目评论或只标记本地完成。

验收标准：

- 用户不用主动扫项目列表，也能看到与自己相关的待行动项目项。
- 写项目评论或状态前必须 review。

### 10.3 GitLab MR review 请求

1. 私有 GitLab 中一个 MR 请求用户 review。
2. Pharos 创建或更新工作请求。
3. 系统读取 MR 描述、diff 摘要、discussion、pipeline 状态。
4. 系统生成 review 摘要、风险点、测试缺口和评论草稿。
5. 用户在 Review Gate 中编辑后批准。
6. Pharos 写入 GitLab comment 或 review discussion。
7. 请求完成，证据保留。

验收标准：

- MR 更新后同一请求被更新，而不是重复创建。
- 自动 review 只读分析可以自动做。
- 写 GitLab 评论必须经过用户批准。

### 10.4 飞书文档评论请求

1. 文档评论中有人 @ 用户请求确认。
2. Pharos 创建或更新工作请求。
3. 系统读取相关段落和评论上下文。
4. 系统总结对方要确认的问题和相关背景。
5. 系统生成文档评论草稿。
6. 用户批准后写入评论，或归档/稍后。

验收标准：

- 文档可以作为请求来源，也可以作为其他请求的上下文证据。
- MVP 不自动修改文档正文。

### 10.5 手动捕获请求

1. 用户在菜单栏、主窗口或 CLI 中输入一段文字/链接。
2. Pharos 创建工作请求。
3. 系统自动整理标题、类型、优先级、建议下一步。
4. 用户后续在 Today 中处理。

验收标准：

- 手动捕获到 Today 可见不超过 3 秒。
- 手动请求同样走 triage、context、execution、review 的统一模型。

## 11. 页面需求

### 11.1 Today

默认首页，回答：现在什么需要我的注意？

分区：

1. Needs Decision：需要用户拍板的事项。
2. Needs Input：缺上下文、缺权限或缺选择的事项。
3. Watching：Pharos 正在看住、整理或等待的事项。
4. Handled：今日已经处理完成的事项。
5. Noise summary：被过滤或归档的信号摘要。

辅助筛选：

- source。
- risk level。
- priority。
- internal status。
- updated time。

### 11.2 Request Detail

请求详情页，回答：这是什么？为什么进来？证据是什么？我批准后会发生什么？

核心区域：

1. Header：标题、来源、用户表层分组、内部状态、优先级、风险。
2. Summary：请求摘要、进入原因和当前需要用户做的决定。
3. Context：上下文包摘要，区分事实、推断和待确认点。
4. Proposed Actions：建议动作、草稿、写入目标和风险说明。
5. Review Gate：批准、编辑、拒绝、补上下文、稍后处理、归档等操作。
6. Evidence：来源证据和系统判断依据。
7. Timeline：执行过程、用户决策、写回结果。

### 11.3 Sources

数据源管理页。

需要支持：

- 开关 source。
- 配置收集范围。
- 查看连接状态。
- 查看最近同步错误。
- 设置写入权限。
- 设置 source 默认优先级和通知策略。

### 11.4 Rules

规则页。

MVP 支持简单规则，不做复杂工作流编辑器。

例子：

- 某飞书群默认 low。
- 包含“线上”“阻塞”“发布”的消息默认 high。
- 某 GitLab repo 的 MR 默认进入 Today。
- 某类文档评论默认只归档。
- 所有 GitLab 写入必须 review。

### 11.5 Metrics

效果验证页。

展示：

- 今日概览。
- 7 天趋势。
- source 质量。
- 自动推进质量。
- review 质量。
- 误报与漏报。
- 未审批写入安全指标。

## 12. 扩展模型

MVP 必须给后续扩展留下清晰边界。

### 12.1 Source Adapter 扩展

任何新数据源都应该只负责：

1. 把外部事件转成 Source Signal。
2. 提供按需读取上下文的能力。
3. 提供受控写回能力。
4. 声明自己支持的事件类型、上下文类型、写入类型和权限边界。

新 source 不应该直接绕过 Pharos 的 triage、risk、review、evidence 模型。

### 12.2 Skill 扩展

任何新 skill 都应该只负责处理某类工作请求或上下文，不应该拥有最终授权权力。

Skill 可以产生：

- 摘要。
- 判断。
- 草稿。
- 建议动作。
- 证据。
- 需要用户确认的问题。

Skill 不可以绕过 Review Gate 执行外部写入。

### 12.3 Policy / Gate 独立性

风险策略和审批门禁必须独立于 source 和 skill。

理由：

- source 可能有 bug。
- skill 可能误判。
- 模型可能幻觉。
- 外部系统写入需要统一治理。

因此，外部写入前必须由统一 policy 检查：

1. 这个动作属于什么风险等级。
2. 是否需要 approval。
3. 是否已有 approval。
4. approval 是否匹配当前动作内容。
5. 写入目标是否在白名单内。

### 12.4 Future Integration

未来可接入：

- `vc`：作为 CLI bridge 或本地自动化入口。
- `skills`：作为 skill registry。
- `intent-to-evidence`：作为更强的意图、计划、证据、验收链。
- `nyx`：作为聊天/agent runtime 经验来源。
- 更多工作系统：邮件、日历、告警、Sentry、Jira、Slack、Notion 等。

但这些都不是 MVP 的强依赖。

## 13. 安全与权限原则

### 13.1 默认策略

1. 默认只读。
2. 默认不写外部系统。
3. 默认外部写入需要 review。
4. 默认高影响动作只给建议。
5. 默认保留证据和执行记录。
6. 默认 source 失败隔离。

### 13.2 不变量

这些规则不能被业务便利性打破：

1. 所有外部写入必须有用户批准记录。
2. 所有进入 Today 的请求必须有进入原因。
3. 所有等待 review 的请求必须有证据。
4. 所有建议动作必须有风险等级。
5. source adapter 失败不能拖垮整个 app。
6. skill 不能绕过 Review Gate。
7. 用户必须知道批准后会写到哪里。
8. 用户必须能标记误报。
9. 请求不能静默删除，只能归档。
10. 未审批外部写入次数必须永远为 0。

## 14. MVP 验收标准总表

### 14.1 功能验收

| 模块 | MVP 必须达到 |
|---|---|
| macOS App | 可启动，有主窗口、菜单栏入口、基础导航。 |
| 本地持久化 | 重启后请求、证据、执行记录、设置不丢失。 |
| 手动捕获 | 能从 UI 或 CLI 创建工作请求。 |
| 飞书聊天 | 能从配置范围内的 @、关键词或转发消息创建/更新请求。 |
| 飞书项目 | 能从分配、评论、状态变化中创建/更新请求。 |
| 私有 GitLab | 能从 MR/Issue/review/mention 中创建/更新请求。 |
| 飞书文档 | 能从文档评论、@ 或手动链接创建请求，也能作为上下文来源。 |
| Triage | 每个 Today 请求都有类型、优先级、风险、进入原因。 |
| 去重合并 | 同一事项不会重复生成多张 active 请求。 |
| 上下文整理 | 请求详情能展示关键上下文和摘要。 |
| 自动执行 | 至少 3 类请求能自动推进到草稿或建议动作。 |
| Review Gate | L3 动作必须等待用户批准，支持编辑后批准。 |
| 受控写回 | 批准后能写飞书消息、飞书项目评论、GitLab comment、飞书文档评论中的至少 2 类。 |
| Evidence | 等待 review 的请求都有来源证据、上下文证据、系统判断。 |
| Timeline | 每次执行过程可追溯。 |
| Rules | 支持基础规则和误报反馈。 |
| Metrics | 可查看 7 天本地指标并导出。 |
| Error Handling | source 或 skill 失败可见、可重试、不崩溃。 |

### 14.2 安全验收

| 规则 | 验收方式 |
|---|---|
| 无 approval 不外部写入 | 构造 L3 动作，未批准时执行按钮不可用或执行被拦截。 |
| 编辑后批准写编辑内容 | 用户修改草稿后批准，外部系统收到的是修改后的内容。 |
| L4/L5 不执行 | 构造代码提交、合并、发布等高风险动作，只允许生成建议。 |
| 写入可追溯 | 所有外部写入能在 Timeline 中找到 approval、内容、目标和结果。 |
| source 失败隔离 | 断开某个 source，其它 source 和本地请求仍可用。 |
| 敏感信息保护 | token、secret 不出现在 UI、日志、metrics 导出中。 |

### 14.3 体验验收

| 指标 | MVP 及格线 |
|---|---|
| 每日有效请求 | 连续 10 个工作日，平均每天 >= 5 个有效请求。 |
| 垃圾请求比例 | 第 1 周 <= 30%，第 2 周 <= 20%。 |
| 高优先级漏报 | 10 个工作日内不漏掉关键 @、关键项目阻塞、关键 MR review。 |
| 自动推进率 | 至少 40% 有效请求能自动推进到摘要、草稿或建议动作。 |
| Review 可判断率 | 至少 70% Ready for Review 请求的证据足够用户做决定。 |
| 草稿可用率 | 可执行草稿中，批准或编辑后批准比例 >= 40%。 |
| 审批效率 | 至少 50% 被批准请求比从零处理更快。 |
| 未审批外部写入 | 永远为 0。 |
| 每日使用 | 连续 10 个工作日每天至少打开一次 Today。 |

## 15. 效果验证方案

### 15.1 Dogfood 周期

MVP 完成后进行 10 个工作日 dogfood。

### 15.2 上线前基线

上线前手动记录 3 个工作日：

1. 每天打开飞书聊天、飞书项目、GitLab、飞书文档的次数。
2. 每天处理多少个 @、项目评论、MR review、文档评论。
3. 每天漏掉或延迟处理的重要事项。
4. 每天花在翻上下文、找链接、整理回复上的时间估算。
5. 每天真正需要用户判断的事项数量。

### 15.3 上线后自动记录

Pharos 自动记录：

1. source signal 数。
2. 工作请求数。
3. 进入 Today 数。
4. Ready for Review 数。
5. 自动推进数。
6. approval / edit approval / reject 数。
7. false positive 数。
8. archived 数。
9. 处理耗时。
10. 外部写入次数。
11. 未审批外部写入次数。

### 15.4 每日主观复盘

每天结束时填写 5 个问题：

```text
今天 Pharos 是否减少了我扫系统/翻上下文的次数？0-5
今天 Pharos 是否让我少漏事？0-5
今天 Pharos 是否制造了额外负担？0-5
今天最有价值的一张请求是什么？
今天最烦或最没用的一张请求是什么？
```

### 15.5 继续投入条件

满足以下条件，说明 MVP 值得继续：

1. 连续 10 个工作日每天使用。
2. 平均每天 >= 5 个有效请求。
3. 至少 3 类请求能稳定自动推进。
4. 第 2 周垃圾请求比例 <= 20%。
5. 未审批外部写入为 0。
6. 用户主观认为上下文切换减少，平均分 >= 4/5。
7. 用户开始自然地把工作请求投给 Pharos，而不是回到原系统里手动扫。

### 15.6 收缩或暂停条件

出现以下情况，应收缩范围或暂停扩展：

1. 平均每天有效请求 < 3。
2. 垃圾请求比例持续 > 40%。
3. 用户每天要花大量时间 babysit。
4. Review Gate 证据不足，用户仍要频繁打开外部系统从零判断。
5. source 集成维护成本明显大于节省时间。
6. 外部写入安全边界无法被稳定保证。

## 16. 里程碑建议

### Milestone 0：核心壳与本地请求模型

目标：没有外部 source 也能跑通统一流程。

范围：

- macOS app shell。
- 菜单栏入口。
- 本地持久化。
- 手动捕获。
- 工作请求状态流转。
- Request Detail。
- Review Gate 的本地动作。

验收：

- 用户能手动创建请求。
- 系统能整理请求、生成建议动作。
- 请求能进入 Ready for Review。
- 用户能批准、编辑后批准、拒绝、归档。
- 重启后数据不丢。

### Milestone 1：四类 source 的最小读取接入

目标：真实工作信号能进入 Pharos。

范围：

- 飞书聊天读取。
- 飞书项目读取。
- 私有 GitLab 读取。
- 飞书文档读取。
- Source 设置和连接状态。
- source 失败隔离。

验收：

- 四类 source 都能用测试数据创建或更新工作请求。
- source 失败不会影响其他 source。
- 每个请求都有原始来源和进入原因。

### Milestone 2：需求识别、上下文整理、自动推进

目标：Pharos 不只是收件箱，而是能推进请求。

范围：

- Triage。
- 去重合并。
- 上下文包。
- 内置技能。
- 草稿和建议动作。
- Evidence。
- Timeline。

验收：

- 至少 3 类请求能自动推进到摘要、草稿或建议动作。
- Ready for Review 请求有足够证据。
- 用户可以要求补上下文并重新运行。

### Milestone 3：受控写回

目标：跑通“review 后继续执行”。

范围：

- 飞书聊天回复写回。
- 飞书项目评论写回。
- GitLab comment 写回。
- 飞书文档评论写回。
- approval 记录。
- 写入结果证据。

验收：

- 至少 2 类外部写回跑通。
- 未批准前无法写入。
- 编辑后批准写入编辑后的内容。
- 所有写入可追溯。

### Milestone 4：Dogfood 指标与收敛

目标：判断 MVP 是否有效。

范围：

- Metrics 页面。
- JSON / Markdown 导出。
- 误报/漏报反馈。
- 7 天趋势。
- Dogfood 复盘模板。

验收：

- 能跑 10 个工作日 dogfood。
- 能根据指标判断继续扩展、收缩范围或重做 triage。

## 20. PRD 结论

Pharos MVP 的核心不是某个具体集成，也不是某个单点自动化能力。

它要先证明一套稳定模型：

```text
工作信号可以被收集。
工作信号可以被识别成请求。
请求可以被整理和补上下文。
低风险步骤可以自动推进。
关键动作可以停在 Review Gate。
用户批准后可以继续执行。
整个过程可以被追溯、验证和改进。
```

只要这套闭环成立，后续新增 source、skill、第三方插件、个人事务、团队协作，都只是给驾驶舱增加仪表、信号和执行器，而不是重新定义产品。
